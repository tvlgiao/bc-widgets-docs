    Credit Promotion System Design¶

    Overview¶

    One-time free credit gift for all users (old + new). Extensible for future promotions. Controlled via environment variables. Grant happens on /load (when user opens admin app).

    Data Model¶

    User model — new field¶

    creditPromotions: [{
    code: String,            // unique promo identifier, e.g. 'welcome_5usd'
    amountCents: Number,     // amount granted, e.g. 500
    claimedAt: Date,         // when credit was granted
    noticeDismissedAt: Date, // null = chatbot notice not yet dismissed
}]

    ApiUsageLog — new enum value¶

    Add 'credit_promo' to subscriptionEvent enum. Log entry includes: - storeHash, subscriptionEvent: 'credit_promo' - amountCents, balanceBeforeCents, balanceAfterCents - Promo code stored via existing fields or log context

    Environment Variables¶

    CREDIT_PROMO_CODE=welcome_5usd    # empty string or unset = feature disabled
CREDIT_PROMO_CENTS=500            # amount in cents ($5 = 500)

    Server Changes¶

    /load endpoint (server/routers/load.js)¶

    After verifyPayload + verifyUser, before building response:

    1. Read CREDIT_PROMO_CODE and CREDIT_PROMO_CENTS from env
    2. If code is empty or cents <= 0 → skip (feature disabled)
    3. Check if user.creditPromotions already contains entry with matching code
    4. If not claimed → atomic findOneAndUpdate: 
      User.findOneAndUpdate(
    {
        _id: user._id,
        'creditPromotions.code': { $ne: PROMO_CODE },
    },
    {
        $push: {
            creditPromotions: {
                code: PROMO_CODE,
                amountCents: PROMO_CENTS,
                claimedAt: new Date(),
                noticeDismissedAt: null,
            }
        },
        $inc: { creditBalance: PROMO_CENTS },
    },
    { new: true }
);
    5. If update succeeded → add creditPromoAwarded: { code, amountCents } to response
    6. Fire-and-forget log to ApiUsageLog with subscriptionEvent: 'credit_promo'

    Chatbot notice dismiss endpoint¶

    New route or extend existing /ai routes:

    POST /ai/dismiss-promo-notice
Body: { store_hash, code }

    Sets noticeDismissedAt on the matching creditPromotions entry: 

    User.findOneAndUpdate(
    { context: `stores/${storeHash}`, 'creditPromotions.code': code },
    { $set: { 'creditPromotions.$.noticeDismissedAt': new Date() } }
);

    Chatbot promo status¶

    Extend existing chatbot data flow (e.g. subscription status check) to also return: 

    {
    "activePromo": {
        "code": "welcome_5usd",
        "amountCents": 500,
        "noticeDismissed": false
    }
}

    Only returned when a matching promo exists and noticeDismissedAt === null.

    Client Admin App Changes (client/src/App.js)¶

    Redux state¶

    fetchAuth response now may include creditPromoAwarded. Store in auth slice.

    Gift Banner¶

    When creditPromoAwarded is present in load response: - Show a top banner (styled differently from PromoBanner — gift/celebration theme) - Content: "You've received $X.XX free credits! Try our AI HTML Generator." - CTA "Use Now" button → setActiveTab('widgets') + setHighlightWidgetName('AI HTML Generator | PapaThemes') - Dismiss (X) button hides banner (session only, no persistence needed — it only shows on first load when credit is granted)

    Chatbot Changes (client/widgets/)¶

    Inline notice in chatbox¶

    When chatbot sidebar opens and state.messages.length === 0: 1. Fetch promo status from server (piggyback on existing subscription check or separate call) 2. If activePromo exists and !noticeDismissed: - Show inline notice via addInlineNotice(): "You have $X free credits! Start chatting to create HTML sections with AI." 3. When user sends first message: - Call POST /ai/dismiss-promo-notice (fire-and-forget) - Remove the inline notice element from DOM

    No localStorage¶

    All state is DB-backed. Once noticeDismissedAt is set, no chatbox instance will show the notice again.

    Flow Diagram¶

    User opens admin app
  → POST /load
  → Server checks CREDIT_PROMO_CODE env
  → User.creditPromotions has code?
    ├─ No  → atomic grant + log → response includes creditPromoAwarded
    │        → Client shows gift banner with "Use Now" CTA
    └─ Yes → normal response

User opens chatbot sidebar
  → messages.length === 0?
    ├─ Yes → fetch promo status
    │        → activePromo && !noticeDismissed?
    │          ├─ Yes → show inline notice
    │          └─ No  → nothing
    └─ No  → nothing

User sends first message in chatbot
  → dismiss promo notice (fire-and-forget API call)
  → remove notice from DOM
  → all future chatbox opens: noticeDismissedAt is set → no notice

    Future Extensibility¶

    To add a new promotion: 1. Change env vars: CREDIT_PROMO_CODE=summer_2026, CREDIT_PROMO_CENTS=300 2. All users who haven't claimed summer_2026 will get it on next app load 3. Previous promotions remain in creditPromotions array (audit trail) 4. Each promo has independent notice dismiss state
