    Compact Conversation Feature - Requirements & Acceptance Criteria¶

    1. Problem Statement¶

    As conversations grow longer, every subsequent message re-sends the entire conversation history to the Claude API. On a 20-message conversation, the input tokens can reach 100K+, costing the merchant significant credits per message. Today's auto-compaction only triggers when the context window is nearly full (~180K tokens), meaning merchants pay for inflated input tokens long before that threshold.

    Merchant pain point: "I'm 15 messages in, my context stats show 80K tokens used, and each new message costs me ~$0.24 in input alone. I want to keep my conversation but stop paying for the full history on every call."

    2. Target Persona¶

    • Primary: Power users who build complex widgets over 10-30+ message sessions
    • Secondary: Cost-conscious merchants on small credit balances who want to extend their budget
    • Technical skill: Low to medium (the UI must make it obvious what compaction does)

    3. Value Proposition¶

    Aspect Current State After Compact
    Input tokens per message (20-msg session) ~80,000-120,000 ~5,000-15,000
    Cost per follow-up message (Sonnet) ~\(0.24-\)0.36 ~\(0.015-\)0.045
    Token savings None (until auto-compact) 70-90% reduction on input tokens
    Conversation context quality Full verbatim history Summarized key decisions + current code + recent messages

    Core differentiator from existing auto-compact: User-initiated, happens much earlier in the conversation, and the user retains full visibility of their chat history in the UI.

    4. How It Differs from Existing Auto-Compaction¶

    The server already has compactMessages() in ai-token-manager.js which auto-triggers when context exceeds ~180K tokens. The new feature differs:

    Dimension Existing Auto-Compact New User-Initiated Compact
    Trigger Automatic when near context limit User clicks a button
    Timing Very late (80%+ context used) Any time after first AI response
    Persistence Transient (only for that API call) Persisted -- subsequent messages use compacted version
    UI feedback None (invisible to user) Inline notice + context stats update
    Full history Lost (messages truncated/summarized in state) Preserved in UI and in DB

    5. User Stories & Acceptance Criteria¶

    US-1: Compact Button Appearance¶

    As a merchant using the AI chatbot, I want to see a "Compact Conversation" tip after each AI response completes, so that I know I can reduce my token costs at any point.

    Acceptance Criteria: 1. After each completed assistant message (not during streaming), a hint element appears below the message in the chat flow. 2. The hint displays: a compress/shrink icon + "Compact conversation to save tokens" (or similar copy). 3. The hint is styled as a subtle, non-intrusive tip (similar to the existing "Continue" button pattern -- light background, small text, icon + text + action button). 4. The hint does NOT appear after the very first assistant response (nothing to compact yet -- need at least 2 user+assistant exchanges). 5. The hint does NOT appear if the conversation is already compacted and no new messages have been added since. 6. The hint disappears when the user starts typing a new message or when they click compact. 7. If the conversation is already compacted and new messages are added, the hint reappears (re-compaction is possible).

    US-2: Compact Action Execution¶

    As a merchant, I want clicking "Compact" to summarize my conversation history via the Claude API, so that future messages are cheaper without losing context quality.

    Acceptance Criteria: 1. Clicking the compact button sends a request to a new server endpoint: POST /ai/compact-conversation. 2. The server uses claude-haiku-4-5 (cheapest model) to generate the summary, regardless of the user's selected model. 3. The summary prompt instructs Claude to preserve: - Key design decisions and requirements discussed - Current widget code (the latest <generated_code> block) - Any referenced image descriptions - User preferences expressed during the conversation - Error resolutions and fixes applied 4. The compacted result is a single [{ role: 'user', content: '<conversation_summary>...' }, { role: 'assistant', content: 'I understand...' }] message pair (matching existing hybridCompact format). 5. The compact API call costs credits (charged like any other AI call). The cost is shown to the user before confirming. 6. During compaction, a "Compacting conversation..." indicator appears (reuse thinking dots pattern). 7. If compaction fails (API error, insufficient credits), show an error notification and leave the conversation unchanged.

    US-3: Post-Compact Message Sending¶

    As a merchant, I want messages sent after compacting to use ONLY the compacted summary (not full history) as context, so that I actually save tokens/credits.

    Acceptance Criteria: 1. After compaction, state.compactedMessages holds the summary + any new messages added after compaction. 2. When sending to /ai/chat-pb, the request body's messages field contains ONLY state.compactedMessages (not state.messages). 3. The full state.messages array remains unchanged and is still used for UI display. 4. The /context stats endpoint reflects the reduced token count post-compaction. 5. Token usage reported after compaction should be significantly lower than before (testable via /context command).

    US-4: Full History Preservation in UI¶

    As a merchant, I want to still see my complete conversation history in the chat UI after compacting, so that I don't lose any information visually.

    Acceptance Criteria: 1. After compaction, all messages remain visible in the chat container -- no messages are removed or collapsed. 2. An inline notice appears in the chat flow at the compaction point: "Conversation compacted -- future messages will use a summarized context to save tokens." 3. Closing and reopening the chat widget shows the full message history (loaded from state.messages / DB). 4. The version history (code versions) is unaffected by compaction.

    US-5: Persistence of Compacted State¶

    As a merchant, I want the compacted state to persist across chat sessions, so that reopening the chat doesn't lose the compaction benefit.

    Acceptance Criteria: 1. The ChatHistory model stores a new field: compactedMessages (array, same structure as messages). 2. The ChatHistory model stores compactedAt (Date) indicating when last compaction occurred. 3. On saveChatHistory(), both messages (full) and compactedMessages (compacted) are saved. 4. On loadChatHistory(), if compactedMessages exists and is non-empty, it is loaded into state.compactedMessages. 5. For sessionStorage fallback (non-subscribers), compactedMessages is also saved/loaded.

    US-6: Re-Compaction and New Messages After Compact¶

    As a merchant, I want to be able to compact again after adding more messages to a previously compacted conversation, so that I can keep costs low throughout a long session.

    Acceptance Criteria: 1. After compaction, new user/assistant messages are appended to BOTH state.messages (full history) and state.compactedMessages. 2. If the user clicks compact again, a NEW compaction runs on state.compactedMessages (not full history -- this is cheaper because compactedMessages is already small + new messages). 3. The compaction hint reappears after 2+ new message exchanges post-compact. 4. There is no limit on how many times the user can compact. 5. Each re-compaction replaces the previous compactedMessages with a new summary.

    US-7: Context Stats Reflect Compaction¶

    As a merchant, I want the /context command to show whether my conversation is compacted and the token savings, so that I can verify the cost reduction.

    Acceptance Criteria: 1. The context stats panel shows a "Compaction" section with: - Status: "Active" (green) or "Not compacted" - Original message count vs. API message count - Estimated token savings (original tokens - compacted tokens) - Last compacted timestamp 2. The utilization bar reflects the compacted token count (not the full history count).

    6. Data Model Changes¶

    ChatHistory Model (MongoDB)¶

    Add to server/models/ChatHistory.js:

    // Compacted conversation for API calls (saves tokens)
compactedMessages: [{
    role: {
        type: String,
        enum: ['user', 'assistant'],
        required: true
    },
    content: {
        type: mongoose.Schema.Types.Mixed,
        required: true
    }
}],
compactedAt: {
    type: Date,
    default: null
}

    Client State¶

    Add to client/widgets/chatbot/state.js:

    // Compact conversation state
compactedMessages: null,    // Array of messages to send to API (null = use full messages)
compactedAt: null,          // Date of last compaction

    7. New Server Endpoint¶

    POST /ai/compact-conversation¶

    Request: 

    {
    "store_hash": "abc123",
    "request_token": "...",
    "instanceId": "widget-xyz",
    "messages": [...],
    "currentCode": "...",
    "model": "claude-haiku-4-5"
}

    Processing: 1. Verify store_hash and request_token. 2. Check credit balance (estimate compaction cost: ~1K-2K output tokens on Haiku = ~\(0.001-\)0.002). 3. Build summary prompt: 

    Summarize this AI widget-building conversation concisely.
Preserve: key design decisions, user requirements, current code state,
image descriptions referenced, errors encountered and resolved,
user preferences for style/layout. Be thorough but concise.
Current code: [latest generated_code block]
Conversation: [messages as user/assistant pairs, truncated to 500 chars each]
    4. Call Claude Haiku with the summary prompt. 5. Return the compacted messages pair. 6. Charge credits for the API call.

    Response (success): 

    {
    "compactedMessages": [
        { "role": "user", "content": "<conversation_summary>...</conversation_summary>" },
        { "role": "assistant", "content": "I understand the context..." }
    ],
    "usage": {
        "input_tokens": 5000,
        "output_tokens": 1500
    },
    "cost_cents": 0.2
}

    Response (error): 

    {
    "error": "Insufficient credits for compaction"
}

    8. Client-Side Changes¶

    Files to Modify¶

    File Changes
    client/widgets/chatbot/state.js Add compactedMessages, compactedAt to state
    client/widgets/chatbot/send-message.js Use state.compactedMessages (if set) instead of state.messages in API request body
    client/widgets/chatbot/messages.js Render compact hint after assistant messages; render inline compaction notice
    client/widgets/chatbot/api.js Add compactConversation() API call; update saveChatHistory/loadChatHistory to include compacted state
    client/widgets/chatbot/context-stats.js Add compaction section to stats display
    client/widgets/chatbot/styles.js Add styles for compact hint, inline notice
    server/models/ChatHistory.js Add compactedMessages, compactedAt fields
    server/routers/ai.js Add POST /ai/compact-conversation endpoint; modify /ai/chat-pb to accept compacted state

    Key Change in send-message.js¶

    In the sendMessage function, at line ~291-302 where the fetch request is built:

    // BEFORE (current):
body: JSON.stringify({
    ...
    messages: state.messages,
    ...
})

// AFTER:
body: JSON.stringify({
    ...
    messages: state.compactedMessages || state.messages,
    ...
})

    After receiving a response, append the new user+assistant pair to BOTH arrays:

    // Existing: state.messages.push(...)
// New: also push to compactedMessages if it exists
if (state.compactedMessages) {
    state.compactedMessages.push(
        { role: 'user', content: messageContent },
        { role: 'assistant', content: finalText }
    );
}

    9. Edge Cases¶

    Edge Case Handling
    User compacts with only 1 exchange Compact hint hidden; minimum 2 exchanges required
    Compaction API call fails Show error notification, state unchanged, hint remains
    Insufficient credits for compaction Show specific "insufficient credits" message with cost estimate
    User sends message during compaction Queue the message (existing queue system handles this)
    Conversation has images Summary preserves image descriptions but not base64 data; images remain in full history
    User does "New Chat" after compacting Both messages and compactedMessages are cleared
    Non-subscriber (sessionStorage) compactedMessages saved/loaded from sessionStorage alongside messages
    Very short conversation (< 5K tokens) Compact hint still appears but a warning: "Conversation is small -- compaction may not save significant tokens"
    Compacted state loaded but server changes compactedMessages treated as source of truth for API calls until next compact or new chat

    10. Cost Analysis for Merchants¶

    Scenario: 20-message widget-building session on Claude Sonnet¶

    Without compaction: - Messages 1-10: Average ~50K input tokens per call = ~\(0.15/call - Messages 11-20: Average ~100K input tokens per call = ~\)0.30/call - Total input cost: ~$2.25

    With compaction after message 10: - Messages 1-10: Same as above = ~\(0.15/call avg = ~\)1.50 - Compaction call (Haiku): ~5K input + 1.5K output = ~\(0.002 - Messages 11-20: Average ~15K input tokens per call = ~\)0.045/call - Total input cost: ~$1.50 + \(0.002 + ~\)0.45 = ~$1.95

    Savings: ~$0.30 (13%) for one compaction, more with multiple compactions

    For longer sessions (30+ messages) or conversations with images/code, savings compound to 30-50%.

    11. Phased Rollout¶

    Phase 1 (MVP) -- Recommended Start¶

    • Compact button as inline hint after assistant messages
    • Server endpoint using Haiku for summarization
    • Compacted state persisted in DB and sessionStorage
    • Basic inline notice at compaction point
    • Credit charge for compaction call

    Phase 2¶

    • Context stats showing compaction savings
    • "Compact & Continue" option (compact + send queued message in one action)
    • Auto-suggest compaction when context reaches 50% utilization
    • Preview of summary before confirming compaction

    Phase 3¶

    • Auto-compact preferences in admin settings (compact automatically at X% utilization)
    • Compaction quality feedback ("Was this summary accurate?")
    • Smart compaction that preserves more detail for code-heavy conversations

    12. Success Metrics¶

    Metric Target Measurement
    Adoption rate 30% of 10+ message sessions use compact Track compact_conversation events in analytics
    Token savings 50%+ reduction in input tokens for subsequent messages Compare lastActualInputTokens before vs. after compaction
    Credit extension Merchants with compaction get 25%+ more messages per dollar Compare messages-per-credit for compacting vs. non-compacting sessions
    Conversation quality No increase in "the AI forgot context" complaints Monitor follow-up messages that reference earlier context
    Feature cost efficiency Compaction call costs < 5% of tokens saved Compare compaction API cost vs. cumulative input token savings

    13. Technical Risks & Mitigations¶

    Risk Likelihood Impact Mitigation
    Summary loses critical context (e.g., specific CSS requirements) Medium High Include current code in summary prompt; keep last 4 messages verbatim alongside summary
    Compaction cost offsets savings for short conversations Low Low Only show hint after 2+ exchanges; show cost estimate before confirming
    Race condition: user sends message while compaction is in progress Medium Medium Set state.isCompacting flag; queue messages during compaction (reuse existing queue)
    Summary prompt itself is too large (very long conversations) Low Medium Truncate each message to 500 chars in summary prompt (matching existing summarizeMessages pattern)
    Compacted state diverges from full history after many messages Medium Low Re-compaction resolves this; show hint again after 5+ new exchanges

    14. Open Questions for Product Decision¶

    1. Should compaction be free for the first use per session? This could encourage adoption without merchants worrying about paying to save money.
    2. Should we show a preview of the summary before applying? Adds UX complexity but gives merchants confidence. Recommendation: Phase 2.
    3. Should the compact hint auto-dismiss after a timeout? Risk of being annoying vs. being discoverable. Recommendation: Dismiss on next user input, no auto-timeout.
    4. Should compaction preserve the exact latest code block verbatim (not summarized)? Recommendation: Yes -- always include the full current code in the compacted context.
