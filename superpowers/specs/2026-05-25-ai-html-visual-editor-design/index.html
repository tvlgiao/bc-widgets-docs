    AI HTML Widget — Visual Editor (WYSIWYG in Page Builder)¶

    Date: 2026-05-25 Status: Design — awaiting implementation plan Component: AI HTML Generator | PapaThemes widget

    Note: this spec discusses reading and writing the content element's HTML markup via the DOM. All such writes go through the existing widget pipeline, which already renders merchant-authored markup; no new untrusted-input surface is introduced (see §11).

    1. Goal¶

    Let merchants edit AI-generated content directly and visually inside BigCommerce Page Builder — text, spacing (padding/margin), colors, alignment, border, font, font-size, and typography — without chatting. Edits are responsive: the merchant can adjust styles per breakpoint (Desktop / Tablet / Mobile).

    A later phase adds draw/annotate on a page screenshot, sent with a chat message so the AI can update precisely.

    Scope¶

    • Phase 1 (this spec): Visual style editor with a docked right-side panel, class-based editing (edits the element's <style> rule), shared-class warning, and per-breakpoint responsive editing.
    • Phase 2 (separate spec): Draw/annotate on screenshot → AI chat.

    Out of scope (Phase 1)¶

    • Drag-resize handles / direct manipulation gizmos.
    • Multi-element selection.
    • Gradient / background-image picker (basic solid color only).
    • Intra-drag undo (undo relies on the existing version history).
    • Adding/removing/reordering DOM nodes (structure editing) — styling only.

    2. Decisions (locked)¶

    # Decision Choice
    1 Priority Visual editing first; annotate = Phase 2
    2 UI model Docked right-side panel (Webflow / DevTools style)
    3 Where edits are written Edit-by-class — AI output is class-based (every element has a UUID-prefixed class, styles in a <style> block, no inline). Edit the <style> rule for the element's class (base for Desktop, @media for Tablet/Mobile). Inline = rare fallback.
    4 Shared class behavior When a class is reused across repeated elements (e.g. cards), edit the class rule (affects all) + show a "affects N elements" warning badge
    7 Editable elements Only elements reachable via a class. Clicking a class-less descendant selects the nearest ancestor that has a class.
    5 Responsive Included in Phase 1 — per-breakpoint editing (Desktop / Tablet / Mobile)
    6 Mode toggle Toggle inside the existing chatbot sidebar (Chat ⇄ Design)

    3. Architecture — "DOM is the source of truth"¶

    The editor operates on the live rendered DOM inside currentContentEl (the [data-papathemes-ai-html-content] element). It does not maintain a separate parsed AST model.

    Content model (AI output contract)¶

    The AI system prompt (server/lib/ai-system-prompt.js) mandates class-based output: all generated code is wrapped in <generated_code> and contains a <style> block; the markup is a container div.papathemes-ai-widget-<uuid> and every child element carries a UUID-prefixed class (e.g. .papathemes-ai-widget-<uuid>-header). CSS variables are UUID-prefixed. No inline styles are generated.

    Therefore the editor's model is edit-by-class: styling lives in <style> rules keyed by each element's own class. This is the primary (effectively only) write path. Inline styles are a rare fallback (e.g. raw markup a merchant pasted), not the norm.

    • Read display values via getComputedStyle at the active breakpoint (reliable because the active breakpoint auto-follows the live viewport width — see §6); use the custom breakpoint-aware cascade resolver for source resolution and for previewing a non-active breakpoint.
    • Resolve source of each property — find the <style> rule that styles the selected element's class (base rule for Desktop, the matching @media rule for Tablet/Mobile) — via a CSS rule index built from the content's <style> blocks.
    • Write edits by mutating the matching CSSOM rule inside the live <style> element (inline element.style only in the rare no-class fallback).
    • Serialize the content element's markup back to a string (rebuilding any edited <style> element's textContent from its CSSOM rules) on Save.

    Rejected alternative: a separate AST/string model that re-renders the DOM. It requires model↔DOM sync, causes re-render flicker, loses live <script> state, and adds code for no benefit here.

    Why not just getComputedStyle?¶

    getComputedStyle returns the final computed value but not its source — it cannot tell us which <style> rule (base class rule vs a @media override) a value came from, which we need in order to edit the right place. It also only reflects the currently rendered viewport, so it is unreliable for reading values at a non-active breakpoint.

    4. Integration with the existing widget¶

    Reuse How
    Sidebar (Shadow DOM) Add a Design mode toggle to the existing chatbot sidebar (ai-html.chatbot.js). Chat ⇄ Design switch in the sidebar header.
    Persistence Reuse window.BigCommerce.sendWidgetUpdate({ content }, widgetId).
    Version history Reuse version-control.js (addVersion / applyContent).
    Screenshot cache Call screenshot.invalidate() after each committed edit.
    Edit-mode gating Runs only when _.context.isEditorMode (same as chatbot).

    The selection overlay/highlight is drawn on the light DOM (the storefront preview), positioned via getBoundingClientRect, updated on scroll/resize — mirroring the existing positionInterval pattern. The panel UI lives inside the Shadow DOM sidebar.

    Live-edit vs explicit Save (critical)¶

    Edits are applied directly to the live document and are not persisted to the widget config on every change. sendWidgetUpdate (postMessage) runs only when the merchant clicks "Save changes". Purpose: avoid excessive re-render / churn in Page Builder.

    Two distinct operations:

    • applyEdit — live, no persist. Mutate the document in place: change element.style (inline) or the matching CSSOM rule inside the live <style> element (incl. @media rules). The browser repaints instantly; all elements matching an edited class rule update at once. This call does not serialize, does not call sendWidgetUpdate, and does not push a version. It only marks the editor dirty.

    Mutating the live <style> element's CSSOM (rule.style.setProperty(...)) updates the rendered page without touching the widget's stored code — exactly the goal.

    • saveChanges() — on the Save button. Serialize the content element's markup once (rebuilding edited <style> textContent from its CSSOM), then:
    saveChanges()
  → sendWidgetUpdate({ content: serializedHtml }, widgetId)   // single postMessage
  → versionControl.addVersion(serializedHtml)
  → screenshot.invalidate()
  → clear dirty flag
  // does NOT replace the content markup or re-run scripts (DOM already correct)
    • discardChanges() — optional. Restore the content element to the last saved version (rebuild from versionHistory[currentVersionIndex]), clear dirty.

    Note: this path never goes through the existing onContentUpdate, which replaces the content element's markup and re-executes <script> tags — that remains reserved for AI-generated full replacements. Visual edits keep the live DOM/scripts intact.

    Dirty state, Save / Discard¶

    • Track state.editorDirty. Save and Discard buttons (in the Design panel header) are enabled only when dirty; a small "unsaved changes" indicator shows the state.
    • Warn before any action that would lose unsaved edits: switching Chat ⇄ Design, closing the sidebar, or navigating away (beforeunload while in the Page Builder editor).
    • Because versions are pushed only on Save, undo/redo operates at save granularity. Discarding reverts all unsaved edits to the last saved version (per Phase-1 scope — no per-edit undo).

    5. Modules¶

    New directory: client/widgets/chatbot/visual-editor/

    Module Responsibility
    index.js Orchestrator: Chat⇄Design mode, dirty state, Save / Discard, unsaved-changes guards; wires selection → panel → applyEdit (live) → saveChanges.
    selection.js Click capture on contentEl; resolve the clicked target to the nearest element with a class (class-less descendants select their nearest classed ancestor); hover outline + selected highlight overlay; keyboard (Esc deselect, ↑ select parent).
    css-resolver.js Build rule index from <style> blocks; for a selected element + property, find the governing class rule (base or @media) with specificity + !important; count elements matching the selector; breakpoint-aware computed value (see §6).
    breakpoints.js Breakpoint definitions; map to media queries; auto-detect active breakpoint from live width via matchMedia + resize/ResizeObserver (debounced); emit breakpoint-change events to the panel.
    apply.js applyEdit: apply one property change by mutating the matching CSSOM class rule in the live <style> (base or @media; inline element.style only in the no-class fallback); mark dirty; no persist.
    serialize.js Invoked by Save: serialize the content element's markup (rebuild edited <style> textContent incl. @media blocks from CSSOM) into the string for a single sendWidgetUpdate.
    controls/ Panel control widgets: text, box-model (padding/margin 4 sides), color (color + background-color), typography (font-family / size / weight / line-height / letter-spacing), border (width / style / color / radius), alignment (text-align).

    6. Source resolution & responsive cascade¶

    Breakpoints¶

    Three breakpoints: Desktop (base) / Tablet / Mobile. Cascade is desktop-down:

    • Desktop = base styles (no media query).
    • Tablet = @media (max-width: <tablet-max>) overrides.
    • Mobile = @media (max-width: <mobile-max>) overrides.

    Mobile inherits Tablet inherits Desktop unless overridden. Exact px values are defined in breakpoints.js (default Tablet ≤ 1024px, Mobile ≤ 768px — confirm against Page Builder device widths during implementation).

    Active breakpoint auto-follows the live viewport width¶

    The panel does not require the merchant to manually pick a breakpoint. The active breakpoint is detected from the actual rendered width and the panel shows that breakpoint's configuration automatically:

    • Register a MediaQueryList (window.matchMedia) for each threshold (tablet-max, mobile-max) and listen to its change event; also listen to window resize and a ResizeObserver on document.documentElement as a fallback (debounced ~100ms).
    • On change → compute the active breakpoint (matchMedia is preferred over manual width comparison because it matches the exact media-query conditions the CSS actually uses) → update the panel to that breakpoint's configuration → re-read the selected element's values → set the write target accordingly (base vs the matching @media rule).
    • The breakpoint indicator in the panel reflects the auto-detected mode. A manual override (force a specific breakpoint regardless of width) is an optional secondary control; the default behavior is auto-follow.

    Because the active breakpoint always tracks the live width, getComputedStyle is reliable for value reads at the active breakpoint — the cascade resolver (below) is then only needed for source resolution across media rules and for the optional manual preview of a non-active breakpoint.

    This also removes the hard dependency on a Page Builder postMessage API: when Page Builder switches its device preview, the preview resizes and our resize/matchMedia listeners react to the new width.

    Reading the effective value at a breakpoint¶

    css-resolver parses base rules + all @media rules. For a chosen breakpoint width, it computes the effective declaration for a property by applying every rule whose condition matches that width, ordered by specificity then source order (inline wins; !important respected). This is a mini cascade engine for the supported property subset — needed because getComputedStyle only reflects the live viewport.

    Resolving the source to edit¶

    The selected element has its own class (selection guarantees this — §5). For a property at the active breakpoint:

    1. Desktop (base):
    2. Edit the base <style> rule for the element's class. If that rule does not yet declare the property, add the declaration to it.
    3. If the element has multiple classes, target the rule for its own (most specific / element-level) class; if a property is governed by a different matched rule, the resolver picks the cascade winner and edits that rule.
    4. Tablet / Mobile:
    5. Edit the @media (max-width: X) rule that targets the element's class for that breakpoint. If it does not exist yet → create the @media block and/or the class rule, then add the declaration. No class minting needed — the element already has a class.
    6. No-class fallback (rare): if a selected element genuinely has no class (e.g. raw pasted markup with no class anywhere up the tree), Desktop edits write inline; for responsive in this rare case a unique class is minted as a last resort.

    Shared-class warning¶

    The AI reuses one class across repeated elements (e.g. multiple cards share .papathemes-ai-widget-<uuid>-card). When the class rule being edited matches more than one element (contentEl.querySelectorAll(selector).length > 1), show a badge "affects N elements" and edit the class (per Decision #4). Applies per breakpoint.

    Page Builder device sync¶

    Handled implicitly by the live-width auto-detection above: switching Page Builder's device preview resizes the preview, which our resize/matchMedia listeners pick up. No dependency on a Page Builder postMessage device API. (If such a message ever becomes available it can be used as an additional, more immediate signal, but it is not required.)

    7. Data flow¶

    Single edit (live, no persist):

    click element
  → resolver computes current values + source per property (at active breakpoint)
  → panel shows values + "affects N elements" badge when source is a shared class
  → user changes a control
  → apply.js (applyEdit) mutates the element's class rule in the LIVE <style>
    (base class rule for Desktop / @media class rule for Tablet/Mobile)
  → browser repaints instantly (all elements using that class update together)
  → mark editor dirty   // NO serialize, NO sendWidgetUpdate, NO version push

    Save (explicit, on "Save changes"):

    click "Save changes"
  → serialize.js builds the content string (rebuild edited <style> textContent from CSSOM)
  → sendWidgetUpdate({ content }, widgetId)   // single postMessage
  → versionControl.addVersion(content)
  → screenshot.invalidate()
  → clear dirty; chatbot context now reflects the saved content

    Breakpoint-change flow (viewport resize):

    viewport width changes (resize / matchMedia change / PB device switch)
  → debounce
  → breakpoints.js recomputes active breakpoint
  → panel updates to that breakpoint's configuration
  → if an element is selected: re-read its values (getComputedStyle, now matching width)
    and update the write target (base vs the matching @media rule)
  → reposition the selection overlay

    8. Sync between Visual Editor and AI Chat¶

    • AI applies new code → editor rebuilds the rule index, clears selection, and resets dirty (the AI result becomes the new saved baseline via the existing AI path).
    • Unsaved visual edits + AI action conflict: sending an AI chat message (or accepting an AI result) while the editor is dirty would otherwise discard the live edits. Guard: prompt the merchant to Save or Discard the visual edits first, so the saved content matches the live document before the AI reads/overwrites it.
    • Single source while editing: the AI uses the saved content as context. Requiring a Save (or Discard) before an AI action keeps saved == live, avoiding divergence between what the merchant sees and what the AI receives.

    9. Error handling (no silent failures)¶

    • Rule parse / class-resolution failure for a property → fall back to inline override on the element + log a warning. Editor stays usable.
    • Selected element removed from DOM during editing → deselect, hide panel.
    • Save failure (sendWidgetUpdate unavailable / throws) → keep the live DOM changes, stay dirty, surface an inline error in the panel, allow retry; never drop edits silently.
    • @media rule creation failure → fall back to creating a fresh <style> block owned by the editor.

    10. Testing¶

    Jest (unit): - css-resolver: class-rule source detection (base vs @media), specificity ordering, multi-class winner selection, affected-element count, breakpoint cascade value (Desktop/Tablet/Mobile, including inheritance). - selection: class-less descendant resolves to nearest classed ancestor. - serialize: round-trip (parse → edit → serialize) including @media blocks; no data loss; scripts preserved. - apply: base class-rule edit; @media class-rule create-then-edit (no minting); multi-class targeting; no-class fallback writes inline.

    Playwright (E2E) in a mock Page Builder harness: - Select an element, change padding on a shared class rule → all matching elements update + saved on Save. - Click a class-less child → selection lands on the nearest classed ancestor. - Edit at Mobile breakpoint → an @media (max-width: …) rule targeting the element's class is created and Desktop styling is unchanged. - Switch Chat ⇄ Design without losing state; undo/redo across an AI edit and a visual edit. - Live-edit does NOT call sendWidgetUpdate; multiple edits then a single Save triggers exactly one sendWidgetUpdate + one version push. - Discard reverts all unsaved edits to the last saved version and clears dirty. - Attempting an AI action while dirty prompts Save/Discard before the AI overwrites content. - Unsaved-changes guard fires on Chat⇄Design switch, sidebar close, and beforeunload.

    11. Coding-standards notes¶

    • Panel UI strings via lang/*.json (no hard-coded text); follow project ESLint (4-space indent, single quotes, requires/imports at top).
    • Keyboard accessible panel; visible focus; Esc to deselect.
    • This UI runs only in the Page Builder editor (admin-facing), never on the live storefront.
    • bc-sec review item: the widget already injects {{{content}}} (triple-stache) and the chatbot already writes arbitrary merchant markup; the visual editor adds no new injection surface beyond existing behavior, but the CSS resolver parses merchant content — review for parser robustness.

    12. Open items to confirm during planning¶

    1. Exact Tablet/Mobile breakpoint px to match Page Builder device preview widths.
    2. Debounce window + whether ResizeObserver on document.documentElement is needed in addition to matchMedia change + window resize for reliable detection inside the Page Builder preview frame.
    3. Multi-class targeting: when an element has several classes, which rule the panel edits by default (element-level class) and how to surface that to the merchant.
    4. No-class fallback: confirm how often it can occur given the AI contract, and the minting naming scheme for that rare case only.
